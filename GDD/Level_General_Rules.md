# AISokoban 关卡设计通用规则

> 写给不想读代码的人：关卡是怎么来的、怎么测试的、怎么判断难不难的。

---

## 一、关卡长什么样

一个 Sokoban 关卡 = 一个棋盘，棋盘上有四种东西：

| 符号 | 叫什么 | 说明 |
|------|--------|------|
| 空地 | 地板 | 人和箱子可以站 |
| 墙 | 墙 | 谁也穿不过去 |
| 箱子 | 箱子 | 人可以推，一次推一格，不能同时推两个 |
| 人 | 玩家 | 你控制的角色 |

另外还有 **目标点**（棋盘上面的标记位置），**把所有箱子推到所有目标点上就算过关**。

### 例子

一个最简单的 3 行 × 5 列的关卡：

```
🧱🧱🧱🧱🧱
🧱 👨 📦  🧱
🧱🧱🧱🧱🧱
```

人站在 (1,1)，箱子在 (2,1)，目标点在 (1,1)（地板上的标记）。
解法：人往上一步、往右推一步，箱子到目标点 → 过关。

## 二、关卡是怎么造出来的

### 方法 1：逆向拉动法（自动生成）

我们的生成器用了一种叫「逆向拉动」的技术：

1. **先设计骨架**：定好棋盘的大小、墙的位置、目标点的位置
2. **从"已过关"开始**：把箱子放在目标点上（游戏已结束的状态）
3. **倒着玩**：模拟人把箱子从目标点"拉"出来，拉到棋盘上的某个位置
4. **重复拉**：拉够次数后，箱子就散落在棋盘上了
5. **检查**：让 BFS 求解器验证——从现在的状态出发，能不能真的把箱子推回目标点？能就通过

这个过程自动跑很多遍，每次拉的方向不同，最后挑一个最好的。

### 方法 2：社区关卡库

我们有来自全球 Sokoban 社区的 **38,157 个现成关卡**，按箱子数分文件放在 `data/pools/` 里。

从里面挑关卡的标准：
- 箱子数匹配（主线的第20关要6箱，就从 `pool_6box.json` 里挑）
- BFS 求解器验证是可解的
- 选一个难度合适的替换进来

### 写在 JSON 文件里

关卡数据存在 `data/levels.json`，一个关卡长这样（用记事本也能看懂）：

```json
{
  "id": 1,
  "name": "第一关：初识推箱",
  "cols": 5,
  "rows": 3,
  "step_limit": 40,
  "grid": [
    [1, 1, 1, 1, 1],
    [1, 3, 2, 0, 1],
    [1, 1, 1, 1, 1]
  ],
  "targets": [[3, 1]]
}
```

| 字段 | 意思 |
|------|------|
| id | 关卡的编号 |
| name | 名字 |
| cols / rows | 棋盘有多少列、多少行 |
| grid | 棋盘本身，0=空地 1=墙 2=箱子 3=玩家起点 |
| targets | 目标点的坐标，格式 [列, 行]，左上角是 [0,0] |
| step_limit | 步数上限——超过这个步数就算失败 |

## 三、怎么判断一个关卡能不能玩

### BFS 求解器

这是一个自动解推箱子的程序（在 `tools/bfs_solver.py`）：
- 它用 BFS（广度优先搜索）算法穷举所有可能的走法
- 找到从起点到过关的最短路径
- **最短路径的步数**就是这关的"最小步数"

如果 BFS 找不到解法 → 关卡设计有 bug，不能上线。

### "六格死锁"检测

箱子如果被推到墙角——两面墙呈 90 度夹角夹住——就永远推不出来了。
求解器会自动检测这种情况，跳过会导致死锁的走法，不浪费时间去试。

> 注意：只检测**角死锁**，不检测更复杂的死锁（比如隧道死锁），因为那些容易误判。

## 四、怎么判断难度

### 两个核心数字

| 指标 | 含义 | 举个例子 |
|------|------|---------|
| **BFS 步数** | 电脑找到的最短过关步数 | BFS=41：最优解法要 41 步 |
| **步数上限** | 游戏里给你的步数限额 | step_limit=120 |

### 松紧比

步数上限 ÷ BFS 步数 = 松紧比。这个比例决定了一关有多"宽松"：

| 松紧比 | 感觉 | 用在什么时候 |
|--------|------|-------------|
| 10x 以上 | 极其宽松，随便走 | 前3关教学关 |
| 5x~8x | 比较宽松 | 中期关卡 |
| 3x~5x | 适中，需要想一下 | 中后期 |
| 2x~3x | 有挑战，走错几步就危险 | 后期高难关 |
| 2x 以下 | 极紧，几乎要最优解 | 极限挑战关 |

### 难度曲线

20 关的难度不应该忽高忽低。理想曲线应该是：
- **前几关**：BFS 1-5 步，步限很宽松 → 熟悉操作
- **中间**：BFS 10-25 步，适当收紧 → 动脑筋
- **后段**：BFS 20-40 步，比较紧 → 真正挑战
- **最后一关**：BFS 35-50 步，最紧 → 最终 Boss

如果相邻两关的 BFS 步数突然暴涨（比如 12 → 70），玩家会觉得莫名其妙地难。
如果最终关 BFS 比前面的还低，就没有 Boss 战的感觉。

## 五、如何调整难度

### 调步数上限（最快）

不改关卡结构，只改 `step_limit`：
- 太难的关：把步数上限调大，容错空间更多
- 太简单的关：把步数上限调小，逼迫玩家走最优路径（但别调太紧）

### 换关卡（更彻底）

如果一关本身设计有问题（比如 BFS 步数跟前后不搭、布局不合理），直接从社区库里挑一个同箱子数的替换。

### 重新生成

用逆向拉动法生成器跑新关，调整生成参数：
- 墙越多 → 路径限制越大 → 通常更难
- 拉动次数越多 → 箱子离目标越远 → 需要更多步
- 箱子数越多 → 复杂度指数级上升

## 六、日常开发流程

```
想改关卡
  │
  ├─ 调步数 → 改 levels.json 里的 step_limit
  │
  ├─ 换关卡 → 从 data/pools/pool_Nbox.json 里挑
  │           → 复制 grid 和 targets 到 levels.json
  │           → 跑 BFS 验证可解
  │
  └─ 加新关 → 设计墙布局 → 跑生成器
              → BFS 验证
              → 加到 levels.json
                  │
                  ▼
            ✔ 全部 20 关 BFS 可解
            ✔ 难度曲线平滑
            ✔ 最终关最有挑战
```

### 验证命令

```bash
# 验证所有关卡可解并看难度曲线
cd aisokoban
python3 -c "
import json
from tools.bfs_solver import solve
data = json.load(open('data/levels.json'))
for lv in sorted(data['levels'], key=lambda x: x['id']):
    result = solve(lv)
    print(f'第{lv[\"id\"]}关: BFS={result}, 步限={lv[\"step_limit\"]}')
"
```
